API Overview

Complete reference for the Telegram Bot & Mini App backend API

🌐 Base URL

All endpoints are relative to the base URL. Use your Railway URL in production.

http://localhost:3000/api

🔐 Auth Header

All protected routes require the Telegram initData header:

x-init-data: query_id=...&user=...&hash=...

📦 Request Format

  • Content-Type: application/json
  • File uploads: multipart/form-data
  • All amounts are numbers (Float)
  • telegramId is returned as String

🏷️ Route Access Levels

  • PUBLIC — No auth needed
  • USER AUTH — Telegram initData required
  • ADMIN ONLY — role: admin or superadmin

💡 Key Business Rules

  • System must have an active Session for any request to be submitted
  • Each feature can be toggled independently via Config (deposit_enabled, withdrawal_enabled, etc.)
  • Withdrawal/Recovery eligibility: user must be lastInteractedUser on the Game ID AND have an approved deposit in the current session
  • New ID approval automatically creates a GameId record and links it to the requesting user
  • All approve/reject actions increment session and user summary counters atomically
  • OCR is disabled for now — all deposit slips go straight to pending with ocrStatus: skipped

🔐 Authentication

How the Telegram Mini App authenticates with the backend

How It Works

When the Mini App opens, window.Telegram.WebApp.initData contains a signed string from Telegram. Send this in every protected request as the x-init-data header. The backend validates the HMAC-SHA256 signature using the BOT_TOKEN.

// Frontend — send on every API call
const headers = {
  'Content-Type': 'application/json',
  'x-init-data': window.Telegram.WebApp.initData
}

User Registration

Users are NOT auto-created on auth. The bot handles user creation via the /start command. The mini app must call POST /users/me on load to upsert the user record and get their profile back.

Error Responses

  • 401 — Missing or invalid initData
  • 401 — User not found (start the bot first)
  • 403 — User is banned
  • 403 — Admin access required (admin routes)

⚠️ Error Handling

Standard error response format and error codes used across the API

Error Response Format

{ "error": "error_code_or_message" }

Business Logic Error Codes

Code HTTP Description
system_closed 403 System is closed — no active session
feature_disabled 403 Feature toggled off in config
not_eligible 403 No approved deposit in current session
gameid_not_eligible 403 User is not the lastInteractedUser on Game ID
invalid_gameid 404 Game ID does not exist
banned 403 User account is banned
Already actioned 400 Request already approved or rejected

📥 Admin Requests ADMIN ONLY

View and action all request types. All list endpoints support ?status= and ?sessionId= filters.

GET /admin/requests/deposits|withdrawals|recoveries|newid|tickets|transfers List requests by type
Query Parameters
Param Type Description
status string pending | approved | rejected
sessionId number Filter by session ID
💡 All list results include user, gameId, session, and admin objects for display.
POST /admin/requests/deposits/:id/approve Approve a deposit
💡 Send as multipart/form-data to optionally attach a proof slip image.
Request Body (multipart/form-data)
Field Type Required Description
approvedAmount number optional Adjusted amount. Defaults to requestedAmount.
adminNote string optional Admin note sent to user
slip file optional Proof slip to attach and send to user
⚠️ Increments session totalDepositCount/Amount and user allTime + currentSession counters atomically.
POST /admin/requests/deposits/:id/reject Reject a deposit
Request Body (JSON)
Field Type Required Description
adminNote string required Rejection reason — shown to user in bot message
POST /admin/requests/type/:id/approve — reject Same pattern for all other types
Withdrawals, recoveries, newid, and transfers all follow identical approve/reject pattern. Withdrawals and recoveries also accept a slip file on approve for proof attachment. Rejection always requires adminNote.
All Approve/Reject Endpoints
Endpoint Slip on Approve Amount Adjust
deposits/:id/approve ✅ ✅ approvedAmount
withdrawals/:id/approve ✅ ✅ approvedAmount
recoveries/:id/approve ✅ ❌
newid/:id/approve ❌ ❌
transfers/:id/approve ❌ ❌
tickets/:id/status ❌ ❌ (PATCH, not POST)
